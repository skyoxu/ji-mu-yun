# Phase A 公网前安全加固方案

## 1. 文档目的

本文档定义积木云 Phase A 在“允许公网用户使用浏览器访问，并可触发自由对话 / 正式提交反馈 / 原型运行”之前的最小安全加固方案。

本文档目标不是讨论理想化的长期零信任平台，而是回答一个更具体的问题：

- 当前 Phase A 是否适合直接开放给不受信任的公网普通用户？
- 如果不适合，最小需要补哪些安全边界？
- 这些边界在 Windows ECS 单机环境下应如何落地？

本文档默认环境：

- Windows ECS 单机
- Caddy 作为公网入口反向代理
- PhaseA.Platform 作为 Web/API 服务
- Godot / Codex CLI / 本地脚本在服务器本机执行
- 多项目、多账户，但项目数和并发量受控

## 2. 当前安全结论

### 2.1 当前结论

当前 Phase A 适合作为：

- 单管理员原型验证环境
- 小范围可信用户内测环境

当前 Phase A 不应直接视为：

- 面向不受信任公网普通用户的强隔离多租户环境

原因不是“功能会天然越界”，而是“当前安全边界主要依赖应用约束和 Codex CLI 沙箱，不是操作系统级硬隔离”。

### 2.2 当前能力的风险分级

#### A. 自由对话

- 风险等级：中低
- 当前执行模式：`read-only`
- 主要风险：信息泄露、输出污染、提示注入、用户误导
- 主要非风险：正常情况下不应修改服务器文件

#### B. 白名单技能

- 风险等级：中低
- 当前执行模式：`read-only`
- 主要风险：读取范围过大、输出包含敏感内部信息
- 主要非风险：正常情况下不应修改服务器文件

#### C. 7 步可玩原型 / 正式提交反馈

- 风险等级：高
- 当前执行模式：`workspace-write`
- 主要风险：
  - 用户可驱动模型修改当前项目文件
  - 若沙箱存在绕过面，可能越界到其他目录
  - 若运行账户权限过大，潜在影响会扩大

## 3. 需要防的攻击面

上线公网前，必须把风险拆成四层来看。

### 3.1 项目内破坏

攻击者通过“提交反馈”让模型：

- 删除当前项目源码
- 覆盖当前项目资源
- 破坏测试、场景、配置
- 写入恶意内容到项目日志或导出包

这个层面不是“是否可能”，而是“当前就是允许的设计能力的一部分”。因此这里要做的不是完全禁止，而是把破坏范围限制在当前项目内。

### 3.2 跨项目越权

攻击者尝试通过当前项目能力影响：

- 同一账户下的其他项目
- 其他账户项目
- 共享工作目录
- 共享打包目录

这类风险必须依赖目录边界和 OS 权限边界阻断，不能只靠提示词约束。

### 3.3 平台本体破坏

攻击者尝试影响：

- `PhaseA.Platform` 程序文件
- `phase-a-platform.sqlite3`
- Caddy 配置
- 启动脚本
- 全局日志
- 环境变量引用的敏感命令路径

这类风险一旦成功，就不再是“项目损坏”，而是“平台被接管或失真”。

### 3.4 信息泄露

攻击者尝试让模型或接口暴露：

- 绝对路径
- 命令行参数
- 脚本名
- 其他账户项目名或目录结构
- 平台内部错误细节
- Token 引用、网关信息、主机配置

这部分对当前 Phase A 也同样重要，因为浏览器用户可直接看到模型输出。

## 4. 当前安全边界的真实状态

### 4.1 已有边界

当前已有以下正向边界：

- 项目 repo 路径要求位于 `HOSTED_WORKSPACE_ROOT` 下
- 浏览器 API 按账户和项目做元数据读取
- 自由对话与白名单技能采用 `read-only`
- 正式提交反馈仅允许在原型成功后触发
- 单项目 runner lock 已存在
- 当前已补上中断运行与失联运行的自动回收

### 4.2 当前缺口

当前仍缺以下关键边界：

- 没有独立低权限 runner 账户
- 没有按项目目录实施 NTFS ACL 隔离
- 平台程序目录与项目工作目录没有做 OS 写权限隔离
- `workspace-write` 仍然直接授予给本机 Codex 执行流程
- 没有“只允许 patch 应用”的变更网关
- 没有容器或轻量 VM 隔离层

因此当前安全状态是：

- “应用层有边界”
- “系统层隔离不足”

## 5. 最小上线门槛

如果要把 Phase A 放到公网，并允许普通用户使用“正式提交反馈”，最低必须完成以下三项。

### 5.1 独立低权限 runner 账户

必须新增一个专门执行用户任务的 Windows 本地账户，例如：

- `jimuyun-runner`

这个账户只用于：

- Codex CLI 正式提交反馈
- 7 步原型路线中的可写执行段
- 未来任何 `workspace-write` 类型任务

这个账户不得用于：

- 运行 `PhaseA.Platform`
- 运行 Caddy
- 保存平台数据库
- 持有平台配置写权限

### 5.2 项目目录 NTFS ACL 隔离

每个项目目录必须具备独立 ACL 策略。

推荐目录结构：

```text
C:\jimuyun-hosted\
  platform\
    app\
    data\
    logs\
    caddy\
  workspaces\
    <account-id>\
      <project-id>\
        repo\
        runtime\
        meta\
        packages\
```

要求：

- `PhaseA.Platform` 服务账户对 `platform\` 和 `workspaces\` 有管理权限
- `jimuyun-runner` 仅对当前运行项目目录有读写权限
- `jimuyun-runner` 对其他项目目录无权限
- `jimuyun-runner` 对 `platform\app`、`platform\data`、`platform\caddy` 无写权限
- `jimuyun-runner` 对平台数据库文件无读写权限

### 5.3 平台本体与工作区彻底分层

当前建议把以下内容迁出项目主仓库路径附近，单独落盘：

- 平台程序目录
- SQLite 元数据数据库
- Caddyfile
- Caddy 日志
- 平台运行日志
- 管理 token 文件

必须保证：

- 项目工作区不是平台目录的子目录
- 平台目录不是项目工作区的子目录
- 两者 ACL 完全分离

## 6. 推荐的 Windows 角色划分

### 6.1 服务账户

建议至少分成两类账户：

#### 平台服务账户

例如：

- `jimuyun-svc`

职责：

- 运行 `PhaseA.Platform`
- 访问平台数据库
- 管理项目元数据
- 创建项目目录与设置 ACL

权限要求：

- 对 `platform\` 完整控制
- 对 `workspaces\` 有创建目录和设置 ACL 的权限
- 不需要登录桌面

#### 任务执行账户

例如：

- `jimuyun-runner`

职责：

- 执行 `workspace-write` 型任务

权限要求：

- 默认无权访问平台目录
- 默认无权访问其他项目目录
- 仅在任务启动前临时授予当前项目目录写权限

### 6.2 账户间职责原则

- 平台服务账户负责“调度”
- runner 账户负责“执行”
- 平台服务账户不能把自己的高权限直接透传给用户触发的任务

## 7. 目录与权限建议

### 7.1 建议目录

```text
C:\jimuyun-hosted\
  platform\
    app\
    data\
    caddy\
    runtime-logs\
  workspaces\
    <account-id>\
      <project-id>\
        repo\
        runtime\
        meta\
        packages\
```

### 7.2 权限原则

#### `platform\app`

- `jimuyun-svc`：读写
- `jimuyun-runner`：拒绝写

#### `platform\data`

- `jimuyun-svc`：读写
- `jimuyun-runner`：拒绝读写

#### `platform\caddy`

- `jimuyun-svc`：读写
- `jimuyun-runner`：拒绝读写

#### `workspaces\<account>\<project>`

- `jimuyun-svc`：读写
- `jimuyun-runner`：仅当前任务所属项目可读写
- 非所属 runner：拒绝读写

### 7.3 ACL 管理规则

平台在创建项目时应执行：

1. 创建项目目录
2. 移除继承
3. 仅保留管理员与平台服务账户
4. 在任务启动前为 runner 增加当前项目目录权限
5. 任务结束后保留该项目固定 runner 权限，或按策略回收

Phase A 若仍采用“每项目固定 1 个 runner”的策略，ACL 会更简单。

## 8. 提交反馈功能的安全重构方向

### 8.1 当前模式

当前模式是：

- 浏览器提交反馈
- 后端拼 prompt
- 直接调用 Codex CLI
- 以 `workspace-write` 执行
- 模型直接改项目文件

这个模式适合原型效率，不适合高安全公网。

### 8.2 建议模式 A：最小改造

保留 `workspace-write`，但增加：

- 独立 runner 账户
- 项目级 NTFS ACL
- 平台目录隔离
- 任务超时回收
- 审计日志

优点：

- 改动较小
- 能较快上线

缺点：

- 仍依赖 CLI 沙箱
- 安全强度有限

### 8.3 建议模式 B：推荐改造

把“正式提交反馈”改为两阶段：

1. 模型只生成 patch / 文件变更计划
2. 后端按白名单路径应用 patch

白名单规则示例：

- 允许：`repo\Game.Core\**`
- 允许：`repo\Game.Godot\**`
- 允许：`repo\Tests.Godot\**`
- 允许：`repo\docs\prototypes\**`
- 禁止：`repo\.git\**`
- 禁止：`repo\logs\**`
- 禁止：项目根目录外全部路径

优点：

- 明确限制可写文件集
- 能拦截删除、覆盖、越界写入

缺点：

- 实现复杂度更高
- 需要 patch 验证与失败处理

### 8.4 建议模式 C：强隔离模式

把每次“正式提交反馈”放进：

- Windows Sandbox
- 容器
- 轻量 VM

优点：

- 隔离最强

缺点：

- 运维复杂度最高
- 单机资源消耗增大

## 9. 输出与信息泄露防护

### 9.1 浏览器输出要求

模型输出和错误输出不得暴露：

- 绝对路径
- 脚本名
- 具体命令行
- 环境变量名和值
- 服务器用户名
- 其他项目 ID / 账户 ID

### 9.2 后端处理要求

所有面向浏览器的错误消息必须经过统一清洗层。

推荐策略：

- 原始错误只写服务器日志
- 浏览器只返回用户态摘要
- 对模型最终文本再做一次敏感信息清洗

### 9.3 下载与日志

普通用户不应直接下载：

- 平台运行日志
- 原始 Codex stderr
- 含内部路径的未清洗审计文件

普通用户只应下载：

- 项目包
- 与该项目绑定的、已经过清洗的结果日志

## 10. 运行时控制要求

### 10.1 并发控制

必须保留：

- 单项目仅 1 个运行中任务
- 单账户运行中任务上限
- 全局运行中任务上限

### 10.2 超时控制

建议默认：

- 原型路线：120 分钟
- 正式提交反馈：60 分钟
- 普通分析任务：30 分钟

### 10.3 自动回收

必须具备：

- 服务重启后的中断 run 回收
- 长时间无心跳 run 回收
- runner lock 自动释放

## 11. 审计要求

每次可写任务至少记录：

- accountId
- projectId
- runId
- runType
- model
- 启动时间
- 结束时间
- exitCode
- 是否超时
- 是否被恢复回收

审计日志要求：

- 普通用户不可写
- 普通用户不可直接下载原始版本
- 管理员可查

## 12. 上线前最低检查清单

上线公网普通用户前，至少确认：

- [ ] `PhaseA.Platform` 与 runner 使用不同 Windows 账户
- [ ] 平台数据库与项目目录已分层
- [ ] Caddy 配置与平台程序目录对 runner 不可写
- [ ] 当前项目目录 ACL 已按项目隔离
- [ ] 自由对话保持 `read-only`
- [ ] 白名单 skills 保持 `read-only`
- [ ] 正式提交反馈仅能在当前项目目录执行
- [ ] 已启用中断运行恢复
- [ ] 已启用失联运行恢复
- [ ] 浏览器输出已做路径 / 命令 / 环境信息清洗
- [ ] 下载接口只允许当前项目的打包产物
- [ ] 管理 token 不写入 git 跟踪目录

## 13. Phase A 的建议实施顺序

### 第一步：最低风险上线前必做

1. 调整目录结构，分离 `platform` 与 `workspaces`
2. 引入 `jimuyun-svc` 与 `jimuyun-runner`
3. 对项目目录实施 ACL
4. 验证 runner 无法写平台目录

### 第二步：收紧可写执行面

1. 保持自由对话 / skills 为只读
2. 仅对正式提交反馈保留可写
3. 继续完善输出清洗与下载边界

### 第三步：中期安全升级

1. 把正式提交反馈改为 patch 应用模式
2. 为高价值账户启用更强 runner 隔离
3. 评估容器 / 轻量 VM 路线

## 14. 本文档给出的最终建议

如果近期目标是“尽快但相对稳妥地公网试运行”，建议采用：

- 模式 A 作为立即执行方案
- 模式 B 作为 Phase B / Phase C 的安全升级方向

换句话说：

- 现在不要直接把当前实现原样开放给不受信任公网普通用户
- 先完成“独立 runner 账户 + ACL 隔离 + 平台分层”
- 再决定是否公开开放正式提交反馈

如果上述三项还未完成，推荐的临时策略是：

- 公网仅开放登录、项目查看、只读分析能力
- “正式提交反馈”仅对管理员或可信白名单用户开放
